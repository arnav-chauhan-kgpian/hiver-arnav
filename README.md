# AI Email Suggested Response System

This project generates suggested replies to customer-support emails using
retrieval-augmented generation (RAG) and scores each reply with a composite
evaluation framework. An incoming email is embedded, matched against a corpus of
historical email/reply pairs via nearest-neighbour retrieval, and used together
with the top matches to prompt a Groq-hosted LLM. The generated reply is then
evaluated along five weighted quality dimensions by an LLM judge, adjusted by a
set of deterministic checks, and reduced to a single interpretable score on a
0–100 scale. Evaluation — not generation — is the primary focus of the project:
the system is built so that reply quality can be measured, explained, and
aggregated across a dataset rather than merely produced.

---

## Architecture

```
Incoming Email
      │
      ▼
Sentence Embeddings          (all-MiniLM-L6-v2, L2-normalized)
      │
      ▼
FAISS Retrieval              (IndexFlatIP → cosine similarity)
      │
      ▼
Top-3 Similar Emails         (email + reference reply, as grounding)
      │
      ▼
Groq LLM                     (llama-3.3-70b-versatile, temp 0.3)
      │
      ▼
Generated Reply
      │
      ▼
Composite Evaluation         (LLM judge × 5 dimensions + deterministic checks)
      │
      ▼
CSV Report + Summary Statistics
```

Each stage is an independent module under `src/`, wired together by
`src/pipeline.py`. The LLM-dependent stages degrade gracefully: with no
`GROQ_API_KEY` present, the pipeline still runs dataset generation and retrieval
and exits cleanly.

| Stage | Module | Responsibility |
|-------|--------|----------------|
| Dataset generation | `data/generate_dataset.py` | Synthesize the labelled corpus + train/test split |
| Retrieval | `src/retrieval.py` | Embed, index (FAISS), and query historical emails |
| Generation | `src/generator.py` | Build the RAG prompt and call the Groq LLM |
| Evaluation | `src/evaluate.py` | Score replies (LLM judge + deterministic checks) |
| Orchestration | `src/pipeline.py` | Run stages end-to-end and emit reports |

## Dataset

The corpus is **300 synthetic customer-support conversations**, each a
`(customer_email, agent_reply)` pair with metadata, generated by
`data/generate_dataset.py`. Generation is seeded (`random.seed(42)`), so the
dataset is fully reproducible.

**Categories (10, evenly distributed — 30 each):**
`refund_request`, `shipping_delay`, `password_reset`, `billing_issue`,
`subscription_cancellation`, `feature_request`, `bug_report`, `account_access`,
`invoice_request`, `general_inquiry`.

**Per-record schema:** `id`, `category`, `customer_email`, `agent_reply`,
`priority` (low/medium/high), `customer_name`, and optional `order_id`.

**Realism.** Emails vary in greeting, length, tone, and sign-off. Roughly 15% of
emails contain plausible typos; roughly 20% are written by frustrated customers,
which escalates the record's priority by one level. Refund, billing, and shipping
records carry plausible order numbers (`ORD-YYYY-NNNNNN`) that are referenced
consistently in both the email and the reply.

**Split.** An 80/20 train/test split (240 / 60) is produced deterministically.
Retrieval is built over the **train** split only; test emails are never in the
index, so retrieving context for a test email cannot leak its own reference reply.

**Outputs** (under `data/processed/`): `email_dataset.jsonl`,
`email_dataset.csv`, `train.jsonl`, `test.jsonl`.

### Why synthetic data

Real support inboxes are proprietary and contain PII; using them for a portable,
shareable assignment is not viable. Synthetic data lets us control category
balance, guarantee a paired reference reply for every email, inject specific
realism factors (typos, frustration, order numbers) on demand, and reproduce the
exact corpus on any machine.

### Limitations of synthetic data

The trade-off is honest and worth stating:

- **Lower lexical and structural diversity** than real email. Templates produce
  emails that cluster tightly in embedding space (retrieval similarities within a
  category are very close), which flatters retrieval and inflates evaluation
  scores relative to production.
- **Clean, well-formed references.** Real agent replies contain shortcuts,
  internal jargon, and inconsistencies that the model never has to contend with
  here.
- **No true distribution shift.** Test emails are drawn from the same generator
  as train, so generalisation is not genuinely stressed.

These limitations mean the absolute scores here should be read as a demonstration
of the measurement machinery, not as a production quality estimate.

## Response Generation

**Embeddings.** Customer emails are encoded with the
`sentence-transformers/all-MiniLM-L6-v2` model (384-dim). Vectors are
L2-normalized so inner product equals cosine similarity.

**Retrieval.** Embeddings are indexed in a FAISS `IndexFlatIP` (exact,
brute-force inner-product search — appropriate at this corpus size). Both the
embeddings (`outputs/train_embeddings.npy`) and the index (`outputs/faiss.index`)
are cached and reused across runs to avoid recomputation.

**Top-3 examples.** For a query email, the three nearest train emails and their
reference replies are pulled as grounding context. Three is a deliberate balance:
enough to demonstrate the desired tone and structure, few enough to keep the
prompt focused and within a small token budget.

**Prompt engineering.** The system prompt fixes the agent persona and imposes
hard grounding constraints — *never invent refunds, policies, delivery dates,
discounts, or promises; if information is missing, ask for it politely.* The user
prompt presents the incoming email followed by the retrieved
customer/agent example pairs, and instructs the model to return only the final
reply. These constraints are what make groundedness measurable downstream:
the model is explicitly told not to fabricate, and the evaluator explicitly
checks whether it did.

**LLM.** Generation calls Groq's OpenAI-compatible Chat Completions API
(`llama-3.3-70b-versatile`, `temperature=0.3`, `max_tokens=300`). The low
temperature favours consistent, conservative replies over creative ones.

### Why RAG instead of fine-tuning

- **No training loop or labelled fine-tuning budget required**; the corpus is
  consumed as a retrieval index, so adding, removing, or correcting examples is
  an index rebuild rather than a retraining job.
- **Grounding is inspectable.** The exact examples that conditioned a reply are
  known and can be shown, which directly supports the groundedness dimension of
  the evaluation. A fine-tuned model's "reasons" are opaque by comparison.
- **Lower hallucination surface for a support setting**, where correctness and
  policy-adherence matter more than stylistic flair. Retrieval keeps the model
  anchored to real prior replies.
- **Portability.** RAG works with a hosted, general-purpose LLM behind a standard
  API; no model hosting or training infrastructure is needed.

## Evaluation Framework

This is the core of the project.

**Why not BLEU / ROUGE / exact match.** A customer email has many equally good
replies that share little surface text — different orderings, phrasings, and
levels of detail can all be correct. Lexical-overlap metrics penalise valid
paraphrases and reward copying, so they do not measure reply *quality*. They are
therefore explicitly avoided as the primary signal.

Instead the system uses a **composite metric** over five dimensions:

| Dimension | Weight | What it measures |
|-----------|--------|------------------|
| **Relevance** | 30% | Does the reply directly address the customer's actual issue? |
| **Completeness** | 25% | Does it address every important request or question in the email, not just one? |
| **Groundedness** | 20% | Does it avoid inventing policies, refunds, dates, promises, or facts not supported by the email or reference? |
| **Professional Tone** | 15% | Is it professional, empathetic, concise, and grammatically correct? |
| **Actionability** | 10% | Does it clearly state the next step or the specific information being requested? |

**LLM judge.** The primary scorer is an LLM (the same Groq client used for
generation, at `temperature=0` for determinism). It receives the customer email,
the human reference reply, and the generated reply, and returns a JSON object
scoring each dimension **0–10** with a one-sentence justification per dimension.
Scoring each dimension independently keeps the signal interpretable — a low
overall score can be traced to the specific dimension that caused it.

**Deterministic checks.** After the judge, a set of rule-based checks inspect the
generated reply and *slightly* adjust the score (bounded, so they nudge rather
than dominate), each attaching an explanation:

- greeting present / absent
- closing / sign-off present / absent
- excessive length (> 220 words)
- unsupported monetary or refund amounts (a dollar figure not present in the
  email or reference)
- hallucinated concrete dates (weekday or calendar dates not present in context)
- unsupported discounts or promotions
- empty response (hard-fails to a zero score)

These checks catch concrete, mechanically-verifiable failures that an LLM judge
can overlook or rate inconsistently — particularly fabricated numbers and dates,
which are the highest-risk hallucinations in a support setting.

**Final score.** The five dimension scores are combined by their weights, scaled
to **0–100**, and the deterministic adjustment is applied and clamped to
`[0, 100]`. The evaluator returns the overall score, the per-dimension scores,
and the full reasoning (per-dimension justifications plus deterministic-check
notes).

## Example Evaluation

Produced by `python run.py` on the demo query, using the current code.

### Incoming Email

Hi, I ordered a laptop last week but it still hasn't arrived. Can you tell me when it will be delivered?

### Generated Reply

Subject: Re: Delayed Laptop Delivery

Dear Customer,

Thanks for flagging this, and I'm sorry your laptop is running late. To assist you further, could you please provide me with your order number? This will allow me to check on the status of your order and provide you with a more accurate update on the expected delivery date.

I'll do my best to investigate the cause of the delay and get back to you as soon as possible.

Best regards,
Customer Support Agent

### Evaluation Result

Overall Score: 77.5 / 100

| Dimension | Score |
|-----------|------:|
| Relevance | 8.0 |
| Completeness | 6.0 |
| Groundedness | 9.0 |
| Professional Tone | 9.0 |
| Actionability | 7.0 |

### Why it received this score

- **Relevance (8.0):** The reply directly addresses the delayed-laptop issue, but cannot give the delivery date the customer actually asked for.
- **Completeness (6.0):** It requests the order number needed to proceed, but leaves the estimated delivery date and cause of delay unanswered — the lowest dimension, correctly reflecting a partial answer.
- **Groundedness (9.0):** It invents no dates, refunds, or promises; because the query contains no order number, it asks for one instead of fabricating a status. Deterministic checks added no penalty.
- **Professional Tone (9.0):** Professional, empathetic, and concise, with a proper greeting and sign-off (both confirmed by the deterministic checks).
- **Actionability (7.0):** The immediate next step — provide the order number — is clear, but no subsequent timeline is given.

This example illustrates the intended behaviour of the metric: a safe reply that
declines to hallucinate a delivery date scores high on groundedness and tone
while being penalised on completeness and actionability, rather than receiving a
single opaque number.

## Future Work

- **Human evaluation** on a held-out sample to calibrate the LLM judge against
  human ratings.
- **Pairwise preference evaluation** instead of only pointwise 0–10 scoring,
  which is typically more reliable and less sensitive to judge bias.
- **Retrieval improvements** using hybrid semantic + keyword (e.g. BM25) search
  to complement dense embeddings.
- **Fine-tuning** on real customer-support conversations once available, as an
  alternative or complement to RAG.

---

## Why this evaluation reflects real quality

- **Semantic correctness over lexical similarity.** The judge assesses whether
  the reply *means* the right thing and does the right job, rather than whether it
  reuses the reference's words. This matches how a human reviewer would rate a
  support reply.
- **Groundedness as a first-class dimension.** Hallucinated refunds, dates, and
  promises are the most damaging failures in customer support. Making groundedness
  an explicit dimension — reinforced by deterministic checks for fabricated
  numbers and dates — targets that failure mode directly rather than hoping a
  generic quality score captures it.
- **Interpretable, dimension-level scores.** Every score decomposes into five
  labelled components with written justifications, so a result is actionable: you
  can see *why* a reply scored 72 and not 90.
- **LLM judgement plus deterministic validation.** The two are complementary. The
  LLM handles nuance, intent, and tone; the deterministic layer provides
  reproducible guardrails on the specific, checkable facts where LLM judges are
  least reliable.

### Limitations

- **LLM judges are imperfect.** They carry their own biases (e.g. a mild
  preference for longer or more formal replies) and are not perfectly consistent
  run-to-run even at `temperature=0`. The deterministic layer mitigates but does
  not eliminate this.
- **Synthetic data is not production email.** As noted above, scores here reflect
  a clean, self-consistent distribution and will be optimistic relative to real
  inboxes.
- **Single-reference, pointwise scoring.** Each reply is judged in isolation
  against one reference. Future work: human evaluation on a held-out sample to
  calibrate the judge, pairwise/preference judgments (which are typically more
  reliable than absolute 0–10 scores), and inter-rater agreement between multiple
  judge models to quantify judge variance.

## Running

Requires **Python 3.11**. Install dependencies:

```bash
pip install -r requirements.txt
# or, with uv:
uv pip install -r requirements.txt
```

Generate the dataset (writes to `data/processed/`):

```bash
python data/generate_dataset.py
```

Run the full pipeline (dataset → retrieval → generation → evaluation → report):

```bash
python run.py
```

**Environment variables:**

| Variable | Required | Default | Purpose |
|----------|----------|---------|---------|
| `GROQ_API_KEY` | Yes (for generation/evaluation) | — | Groq API key. Never hardcoded; read from the environment only. |
| `GROQ_MODEL` | No | `llama-3.3-70b-versatile` | Overrides the generation and judge model. |
| `EVAL_TEST_LIMIT` | No | full test set (60) | Caps how many test examples are evaluated, for quick runs. |

Without `GROQ_API_KEY`, the pipeline runs dataset generation and retrieval, then
prints a skip message for the LLM stages and exits cleanly — it does not crash.

### Web app (Streamlit)

An interactive frontend (`app.py`) demonstrates the full pipeline for a single
email — retrieval, generation, and the composite evaluation — reusing the same
`src` modules as the CLI. It exposes the API key, model, retrieval count, and
temperature in the sidebar and renders the retrieved examples, generated reply,
per-dimension scores, pipeline latencies, and downloadable outputs.

Run locally:

```bash
streamlit run app.py
```

The Groq API key can be entered in the sidebar or provided via the
`GROQ_API_KEY` environment variable. **On first launch the app checks for the
processed dataset (`train.jsonl`, `test.jsonl`, `email_dataset.jsonl`) and, if
any are missing, generates them automatically** (showing a "Generating dataset
for first launch..." spinner) before building the retrieval index — no manual
`python data/generate_dataset.py` step is required. If automatic generation
fails, the app shows an error instead of crashing.

### Deploying on Streamlit Community Cloud

1. Push this repository to GitHub.
2. On [share.streamlit.io](https://share.streamlit.io), create a new app pointing
   at the repository, branch, and `app.py` as the entry point.
3. Configure the server-side Groq key under the app's **Settings → Secrets**,
   in TOML form:

   ```toml
   GROQ_API_KEY = "gsk_your_key_here"
   # GROQ_MODEL = "llama-3.3-70b-versatile"   # optional
   ```

   Streamlit Community Cloud injects these secrets as environment variables, so
   the app reads the key via `os.getenv("GROQ_API_KEY")` (with an `st.secrets`
   fallback for local `.streamlit/secrets.toml`). For local development, put the
   same TOML in `.streamlit/secrets.toml` (which is gitignored).
4. Deploy. No dataset setup is required: `data/processed/` is gitignored, but on
   first launch the app detects the missing dataset and generates it
   automatically (deterministically, `random.seed(42)`), then downloads the
   embedding model and builds the FAISS index. Subsequent runs reuse the cached
   index and embeddings.

**API key precedence.** On each run the app resolves the key in this order:

1. A key typed into the sidebar **Groq API Key (optional)** field — used only for
   that session, never written to disk.
2. Otherwise the server-configured key (`GROQ_API_KEY` env / Streamlit secret),
   in which case the sidebar shows *"Using server-configured API key."*
3. If neither is present, the app shows a warning and does not call the API.

This means visitors to a deployed app can simply use the server key with no
setup, or temporarily override it with their own key in the sidebar. The key is
never committed, never logged, and never displayed — it is read only from the
sidebar, the environment, or the platform's secrets store.

## Output

- **`outputs/evaluation_results.csv`** — one row per evaluated test example, with
  columns: `customer_email`, `reference_reply`, `generated_reply`,
  `overall_score`, and the five dimension scores (`relevance`, `completeness`,
  `groundedness`, `professional_tone`, `actionability`).
- **Aggregate statistics** printed at the end of a run — average, median,
  minimum, maximum, and standard deviation of the overall score across the
  evaluated test set.
- **Retrieval/index caches** — `outputs/train_embeddings.npy`,
  `outputs/faiss.index`.

- **Claude Code** (Anthropic) was used for implementation assistance during
