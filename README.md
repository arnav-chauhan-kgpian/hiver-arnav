# AI Email Suggested Response System

**Live demo:** https://hiver-arnav.streamlit.app/

A retrieval-augmented system that drafts customer-support email replies and, more
importantly, **evaluates their quality** with an interpretable composite metric.

---

## Problem Statement

Support teams answer the same kinds of emails repeatedly. An assistant that drafts
a suggested reply can save agent time — but only if the drafts are *trustworthy*.
In a support setting the dangerous failure is not an awkward sentence; it is a
**hallucinated fact**: an invented refund amount, a made-up delivery date, a
promised discount that does not exist. A draft that reads well but fabricates a
commitment is worse than no draft at all.

This makes **evaluation the hard part of the problem**, not generation. Producing
plausible text with an LLM is easy; deciding whether a specific reply is
*relevant, complete, grounded, professional, and actionable* — and doing so in a
way a human can inspect and trust — is where the engineering effort belongs. This
project is built around that idea: generation is a means, and a rigorous,
explainable evaluation of the generated reply is the deliverable.

---

## Overall Architecture

```
Incoming Email
      │
      ▼
Sentence Embeddings          (all-MiniLM-L6-v2, 384-dim, L2-normalized)
      │
      ▼
FAISS Retrieval              (IndexFlatIP → cosine similarity)
      │
      ▼
Top-3 Similar Emails         (email + reference reply, as grounding)
      │
      ▼
Groq LLM                     (llama-3.3-70b-versatile, temp 0.3)
      │
      ▼
Generated Reply
      │
      ▼
Composite Evaluation         (LLM judge × 5 dimensions + deterministic checks)
      │
      ▼
CSV Report + Summary Statistics
```

Each stage is an independent, single-responsibility module under `src/`, wired
together by `src/pipeline.py`. The LLM-dependent stages **degrade gracefully**:
with no `GROQ_API_KEY` present, the pipeline still runs dataset generation and
retrieval and exits cleanly rather than crashing.

| Stage | Module | Responsibility |
|-------|--------|----------------|
| Dataset generation | `data/generate_dataset.py` | Synthesize the labelled corpus + deterministic train/test split |
| Retrieval | `src/retrieval.py` | Embed, index (FAISS), cache, and query historical emails |
| Generation | `src/generator.py` | Build the RAG prompt and call the Groq LLM |
| Evaluation | `src/evaluate.py` | Score replies (LLM judge + deterministic checks) |
| Orchestration | `src/pipeline.py` | Run all stages end-to-end and emit reports |
| Web app | `app.py` | Interactive single-email demo (reuses the modules above) |

## Dataset Creation Process

The corpus is **300 synthetic customer-support conversations**, each a
`(customer_email, agent_reply)` pair with metadata, generated by
`data/generate_dataset.py`. Generation is seeded (`random.seed(42)`), so the
dataset — and its train/test split — is fully reproducible byte-for-byte.

**Categories (10, evenly distributed — 30 each):**
`refund_request`, `shipping_delay`, `password_reset`, `billing_issue`,
`subscription_cancellation`, `feature_request`, `bug_report`, `account_access`,
`invoice_request`, `general_inquiry`.

**Per-record schema:** `id`, `category`, `customer_email`, `agent_reply`,
`priority` (low/medium/high), `customer_name`, and optional `order_id`.

**Realism factors.** Each category has its own email/reply templates with
randomized greetings, sign-offs, products, plans, and reasons. On top of that:
~15% of emails contain plausible typos; ~20% are written by frustrated customers
(which escalates the record's priority by one level); and refund/billing/shipping
records carry plausible order numbers (`ORD-YYYY-NNNNNN`) referenced consistently
in both the email and the reply.

**Split.** An 80/20 train/test split (240 / 60) is produced deterministically.
The retrieval index is built over the **train** split only; test emails are never
in the index, so retrieving context for a test email cannot leak that email's own
reference reply into its evaluation.

**Outputs** (under `data/processed/`): `email_dataset.jsonl`,
`email_dataset.csv`, `train.jsonl`, `test.jsonl`.

**Why synthetic.** Real support inboxes are proprietary and contain PII, so they
are unusable for a portable, shareable assignment. Synthetic data lets us control
category balance, guarantee a paired reference reply for every email, inject
specific realism factors on demand, and reproduce the exact corpus on any machine.
Its limitations are stated honestly under [Limitations](#limitations).

## Retrieval Approach

**Embeddings.** Customer emails are encoded with
`sentence-transformers/all-MiniLM-L6-v2` (384-dim). Vectors are L2-normalized so
that inner product equals cosine similarity.

**Index.** Embeddings are indexed in a FAISS `IndexFlatIP` — exact, brute-force
inner-product search. At 240 vectors this is instant and exact; an approximate
index (IVF/HNSW) would add complexity with no benefit at this scale.

**Query.** An incoming email is embedded and normalized the same way, then the
top-k nearest neighbours are returned as `RetrievedExample` objects
(`customer_email`, `agent_reply`, `category`, `similarity_score`), sorted by
descending cosine similarity. The CLI demo retrieves `k=5`; the web app exposes
`k` as a 1–5 slider.

**Caching.** Both the training embeddings (`outputs/train_embeddings.npy`) and the
FAISS index (`outputs/faiss.index`) are cached and reused across runs, so the
model only re-embeds the corpus when the cache is absent.

---

## Generation Approach

**Grounding.** The top-3 retrieved examples (email + reference reply) are injected
into the prompt as few-shot grounding. Three is a deliberate balance: enough to
convey the desired tone and structure, few enough to keep the prompt focused and
cheap.

**Prompt engineering.** The system prompt fixes the agent persona and imposes hard
anti-hallucination constraints — *never invent refunds, policies, delivery dates,
discounts, or promises; if information is missing, ask for it politely.* The user
prompt presents the incoming email, then the retrieved customer/agent pairs, and
instructs the model to return only the final reply. These constraints are what
make groundedness **measurable** downstream: the model is explicitly told not to
fabricate, and the evaluator explicitly checks whether it did.

**Model.** Generation calls Groq's OpenAI-compatible Chat Completions API
(`llama-3.3-70b-versatile`, `temperature=0.3`, `max_tokens=300`). The low
temperature favours consistent, conservative replies over creative ones.

**Why RAG instead of fine-tuning.** RAG needs no training loop — correcting the
corpus is an index rebuild, not a retraining job; the exact examples that
conditioned a reply are known and inspectable, which directly supports the
groundedness dimension; retrieval keeps the model anchored to real prior replies,
lowering hallucination surface in a setting where policy-adherence matters more
than stylistic flair; and it runs against a hosted general-purpose LLM with no
model-hosting infrastructure.

## Evaluation Methodology

**This is the core of the project.**

### Why not BLEU / ROUGE / exact match

A customer email has many equally good replies that share little surface text —
different orderings, phrasings, and levels of detail can all be correct. A reply
that asks for a missing order number and a reply that quotes a policy verbatim can
both be excellent, yet share almost no tokens with each other or with a single
reference. Lexical-overlap metrics (BLEU, ROUGE, exact match) therefore **penalise
valid paraphrases and reward copying** — they measure text similarity, not reply
*quality*. They are explicitly avoided as the primary signal.

### The composite metric

Quality is decomposed into five weighted dimensions, chosen to capture what a human
reviewer actually cares about in a support reply:

| Dimension | Weight | What it measures |
|-----------|--------|------------------|
| **Relevance** | 30% | Does the reply directly address the customer's actual issue? |
| **Completeness** | 25% | Does it address every important request or question, not just one? |
| **Groundedness** | 20% | Does it avoid inventing policies, refunds, dates, promises, or facts not supported by the email or reference? |
| **Professional Tone** | 15% | Is it professional, empathetic, concise, and grammatically correct? |
| **Actionability** | 10% | Does it clearly state the next step or the specific information being requested? |

Relevance and completeness dominate (55% together) because a reply that answers the
wrong question, or only half the question, is the most common real failure.
Groundedness is weighted heavily (20%) and reinforced by deterministic checks
because hallucinated commitments are the most *costly* failure.

### 1. LLM-as-a-judge (primary signal)

The primary scorer is an LLM (the same Groq client used for generation, at
`temperature=0` for determinism, with `response_format=json_object`). It receives
the customer email, the human reference reply, and the generated reply, and returns
a JSON object scoring each dimension **0–10** with a **one-sentence justification
per dimension**. Scoring each dimension independently keeps the signal
interpretable — a low overall score traces to the specific dimension that caused
it, not an opaque number. JSON parsing is defensive (handles code fences / stray
text), and API calls retry with backoff on transient/rate-limit errors.

### 2. Deterministic checks (guardrails)

After the judge, rule-based checks inspect the generated reply and *slightly*
adjust the score (bounded to ±20 points, so they nudge rather than dominate), each
attaching a plain-text explanation:

- **greeting** present/absent (scanned over the first lines, so a leading
  `Subject:` line does not hide a "Dear …" greeting)
- **closing / sign-off** present/absent
- **excessive length** (> 220 words)
- **unsupported monetary amounts** — a dollar figure in the reply not present in
  the email or reference
- **hallucinated concrete dates** — a weekday ("by Monday"), a real month-date
  ("March 15"), or an `MM/DD` date not present in context. Bare month words (the
  modal verb "may", the verb "march") are deliberately *not* flagged, to avoid
  false positives
- **unsupported discounts / promotions** — a discount, coupon, or `%` not present
  in context
- **empty response** — hard-fails to a score of 0

These catch concrete, mechanically-verifiable failures that an LLM judge can
overlook or rate inconsistently — particularly fabricated numbers and dates, the
highest-risk hallucinations in support.

### 3. Final score

The five dimension scores are combined by their weights, scaled to **0–100**, and
the deterministic adjustment is applied and clamped to `[0, 100]`. The evaluator
returns **`overall_score`** (0–100), **`dimension_scores`** (each 0–10), and
**`reasoning`** (per-dimension justifications plus the deterministic-check notes
and net adjustment). Across the test set these overall scores are aggregated into
average, median, min, max, and standard deviation, and written per-example to
`outputs/evaluation_results.csv`.

### Why this metric is appropriate for email response generation

- **Semantic correctness over lexical similarity.** The judge assesses whether the
  reply *means* the right thing and does the right job, matching how a human
  reviewer rates a support reply — not whether it reuses the reference's words.
- **Groundedness is first-class.** Hallucinated refunds, dates, and promises are
  the most damaging failures in support. Making groundedness an explicit dimension,
  reinforced by deterministic checks for fabricated numbers and dates, targets that
  failure mode directly instead of hoping a generic score catches it.
- **Interpretable and actionable.** Every score decomposes into five labelled
  components with written justifications, so you can see *why* a reply scored 77 and
  not 90 — and which dimension to fix.
- **Judgement + validation are complementary.** The LLM handles nuance, intent, and
  tone; the deterministic layer provides reproducible guardrails on the specific,
  checkable facts where LLM judges are least reliable.

### Worked example

Produced by `python run.py` on the demo query, using the current code.

**Incoming email:**

> Hi, I ordered a laptop last week but it still hasn't arrived. Can you tell me when it will be delivered?

**Generated reply:**

> Subject: Re: Delayed Laptop Delivery
>
> Dear Customer,
>
> Thanks for flagging this, and I'm sorry your laptop is running late. To assist you further, could you please provide me with your order number? This will allow me to check on the status of your order and provide you with a more accurate update on the expected delivery date.
>
> I'll do my best to investigate the cause of the delay and get back to you as soon as possible.
>
> Best regards,
> Customer Support Agent

**Evaluation — Overall Score: 77.5 / 100**

| Dimension | Score |
|-----------|------:|
| Relevance | 8.0 |
| Completeness | 6.0 |
| Groundedness | 9.0 |
| Professional Tone | 9.0 |
| Actionability | 7.0 |

- **Relevance (8.0):** Directly addresses the delayed-laptop issue, but cannot give the delivery date the customer asked for.
- **Completeness (6.0):** Requests the order number needed to proceed, but leaves the estimated date and cause unanswered — correctly the lowest dimension.
- **Groundedness (9.0):** Invents no dates, refunds, or promises; with no order number in the query, it asks for one instead of fabricating a status. Deterministic checks added no penalty.
- **Professional Tone (9.0):** Professional, empathetic, concise, with a greeting and sign-off (both confirmed by the deterministic checks).
- **Actionability (7.0):** The immediate next step is clear, but no subsequent timeline is given.

This is the metric behaving as intended: a **safe** reply that declines to
hallucinate a delivery date scores high on groundedness and tone while being
penalised on completeness and actionability — rather than collapsing to a single
opaque number.

## Trade-offs

Design decisions and what each one costs:

- **Synthetic data vs. real inboxes.** Buys reproducibility, PII-safety, balanced
  categories, and guaranteed reference replies; costs realism — templated emails
  cluster tightly in embedding space, which flatters retrieval and inflates
  scores relative to production.
- **RAG vs. fine-tuning.** Buys inspectable grounding and zero training
  infrastructure; costs the tighter stylistic fit a fine-tuned model could give.
- **LLM judge vs. lexical metrics.** Buys semantic, human-aligned scoring; costs
  determinism and money (an API call per evaluation) and introduces judge bias —
  mitigated by the deterministic layer and `temperature=0`.
- **Top-3 grounding.** Buys a focused, cheap prompt; costs recall on rare phrasings
  that might benefit from more context.
- **Deterministic checks vs. a purely learned score.** Buys reproducible guardrails
  on the highest-risk facts (money, dates, discounts); costs some recall — regex
  checks are conservative by design and only *nudge* the score (±20 cap).
- **Exact FAISS index.** Buys exactness and simplicity at this corpus size; would
  need an approximate index only at much larger scale.

---

## Limitations

- **Synthetic data is not production email.** Scores reflect a clean,
  self-consistent distribution and will be optimistic relative to real inboxes;
  test emails share the generator with train, so generalisation is not stressed.
  The absolute numbers demonstrate the measurement machinery, not a production
  quality estimate.
- **LLM judges are imperfect.** They carry biases (e.g. a mild preference for
  longer or more formal replies) and are not perfectly consistent run-to-run even
  at `temperature=0`. The deterministic layer mitigates but does not eliminate
  this.
- **Single-reference, pointwise scoring.** Each reply is judged in isolation against
  one reference, and the deterministic groundedness checks compare against the
  customer email + top-1 reference only (not all retrieved context).

**Future work:** human evaluation on a held-out sample to calibrate the judge;
pairwise/preference judging (more reliable than absolute 0–10 scores); inter-rater
agreement across multiple judge models to quantify judge variance; hybrid
semantic + keyword (BM25) retrieval; and fine-tuning on real conversations once
available.

## How to Run Locally

Requires **Python 3.11**. Install dependencies:

```bash
pip install -r requirements.txt
# or, with uv:
uv pip install -r requirements.txt
```

Generate the dataset (writes to `data/processed/`):

```bash
python data/generate_dataset.py
```

Run the full pipeline (dataset → retrieval → generation → evaluation → report):

```bash
python run.py
```

**Environment variables:**

| Variable | Required | Default | Purpose |
|----------|----------|---------|---------|
| `GROQ_API_KEY` | For generation/evaluation | — | Groq API key. Never hardcoded; read from the environment only. |
| `GROQ_MODEL` | No | `llama-3.3-70b-versatile` | Overrides the generation and judge model. |
| `EVAL_TEST_LIMIT` | No | full test set (60) | Caps how many test examples are evaluated, for quick runs. |

Without `GROQ_API_KEY`, the pipeline runs dataset generation and retrieval, prints
a skip message for the LLM stages, and exits cleanly — it does not crash. `run.py`
is a thin entry point that simply calls `src.pipeline.run()`.

### Interactive web app

`app.py` demonstrates the full pipeline for a single email — retrieval,
generation, and the composite evaluation — reusing the same `src` modules as the
CLI. It exposes the API key, model, retrieval count, and temperature in the
sidebar and renders the retrieved examples, generated reply, per-dimension score
bars, pipeline latencies, and downloadable outputs.

```bash
streamlit run app.py
```

**On first launch the app checks for the processed dataset and, if any of
`train.jsonl` / `test.jsonl` / `email_dataset.jsonl` are missing, generates them
automatically** (with a "Generating dataset for first launch..." spinner) before
building the retrieval index — no manual dataset step is required. If generation
fails, it shows an error instead of crashing.

## Streamlit Deployment

This app is deployed at **https://hiver-arnav.streamlit.app/**. To deploy your own
instance on Streamlit Community Cloud:

1. Push this repository to GitHub.
2. On [share.streamlit.io](https://share.streamlit.io), create a new app pointing
   at the repository, branch `main`, and `app.py` as the entry point. Set **Python
   3.11** under Advanced settings.
3. Configure the server-side Groq key under **Settings → Secrets**, in TOML form
   (string values **must** be quoted):

   ```toml
   GROQ_API_KEY = "gsk_your_key_here"
   # GROQ_MODEL = "llama-3.3-70b-versatile"   # optional
   ```

   Streamlit Community Cloud injects secrets as environment variables, so the app
   reads the key via `os.getenv("GROQ_API_KEY")` (with an `st.secrets` fallback for
   a local `.streamlit/secrets.toml`, which is gitignored).
4. Deploy. **No dataset setup is required** — `data/processed/` is gitignored, but
   on first launch the app generates the dataset automatically (deterministically,
   `random.seed(42)`), downloads the embedding model, and builds the FAISS index.
   Subsequent runs reuse the cached index and embeddings.

**API key precedence.** On each run the app resolves the key in this order:

1. A key typed into the sidebar **Groq API Key (optional)** field — used only for
   that session, never written to disk.
2. Otherwise the server-configured key (`GROQ_API_KEY` env / Streamlit secret), in
   which case the sidebar shows *"Using server-configured API key."*
3. If neither is present, the app shows a warning and does not call the API.

Visitors can use the server key with zero setup, or temporarily override it with
their own key in the sidebar. The key is never committed, never logged, and never
displayed — it is read only from the sidebar, the environment, or the platform's
secrets store.

### Output artifacts

- **`outputs/evaluation_results.csv`** — one row per evaluated test example:
  `customer_email`, `reference_reply`, `generated_reply`, `overall_score`, and the
  five dimension scores.
- **Aggregate statistics** printed at the end of a run — average, median, min, max,
  and standard deviation of the overall score.
- **Caches** — `outputs/train_embeddings.npy`, `outputs/faiss.index`.
